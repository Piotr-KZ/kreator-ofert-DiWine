 SEKCJA 2: KONWERSJA FIRESTORE → POSTGRESQL

🎯 CEL
Przekształcić NoSQL Firestore (dokumenty, kolekcje) na relacyjny PostgreSQL (tabele, foreign keys, JOINs) zachowując:

Integralność danych
Wydajność queries
Multi-tenancy (izolacja organizacji)
Możliwość skalowania


🗄️ FIRESTORE vs POSTGRESQL - FUNDAMENTALNE RÓŻNICE
Firestore (NoSQL):
javascript// Struktura: Collections → Documents → Subcollections
users/
  {userId}/
    email: "jan@example.com"
    subscription: {               // ❌ Embedded object
      stripeId: "sub_123",
      status: "active",
      plan: "pro"
    }
    metadata: {                   // ❌ Nested map
      lastLogin: Timestamp,
      preferences: {...}
    }
Cechy:

✅ Flexible schema (dodaj pole kiedy chcesz)
✅ Nested objects (embedded documents)
✅ Denormalizacja (duplikacja danych OK)
❌ Brak JOINs (trzeba robić N queries)
❌ Brak transakcji między kolekcjami
❌ Trudne complex queries

PostgreSQL (Relacyjna):
sql-- Struktura: Tables → Rows → Columns
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    subscription_id INTEGER REFERENCES subscriptions(id),  -- ✅ Foreign key
    metadata JSONB                                          -- ✅ JSONB dla flexible data
);

CREATE TABLE subscriptions (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id),
    stripe_id VARCHAR(255),
    status VARCHAR(50),
    plan VARCHAR(50)
);
Cechy:

✅ Fixed schema (migracje dla zmian)
✅ JOINs (pobierz dane z wielu tabel)
✅ ACID transactions
✅ Complex queries (GROUP BY, agregacje)
✅ Foreign keys (referential integrity)
⚠️ Normalizacja wymagana (no duplication)


📋 MAPOWANIE: FIRESTORE → POSTGRESQL
1. COLLECTIONS → TABLES
Firestore:
javascript// Collection: users
users/
  {userId}: {
    email: "jan@example.com",
    name: "Jan Kowalski",
    role: "admin",
    createdAt: Timestamp
  }
PostgreSQL:
sqlCREATE TABLE users (
    id SERIAL PRIMARY KEY,
    firebase_uid VARCHAR(128) UNIQUE,  -- Mapowanie starego ID
    email VARCHAR(255) UNIQUE NOT NULL,
    name VARCHAR(255),
    role VARCHAR(50) DEFAULT 'user',
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Index dla szybkiego lookup
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_firebase_uid ON users(firebase_uid);

2. EMBEDDED OBJECTS → SEPARATE TABLES
Firestore (Denormalizacja):
javascriptusers/{userId}: {
  email: "jan@example.com",
  subscription: {              // ❌ Embedded - trudno query
    stripeId: "sub_123",
    status: "active",
    currentPeriodEnd: Timestamp
  }
}
PostgreSQL (Normalizacja):
sql-- Tabela główna
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    -- Brak subscription fields!
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Osobna tabela dla subscriptions
CREATE TABLE subscriptions (
    id SERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    stripe_subscription_id VARCHAR(255) UNIQUE,
    status VARCHAR(50) NOT NULL,
    current_period_start TIMESTAMP WITH TIME ZONE,
    current_period_end TIMESTAMP WITH TIME ZONE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    
    -- Jeden user = jedna aktywna subskrypcja
    CONSTRAINT unique_active_subscription 
        UNIQUE (user_id)
);

CREATE INDEX idx_subscriptions_user_id ON subscriptions(user_id);
CREATE INDEX idx_subscriptions_stripe_id ON subscriptions(stripe_subscription_id);
Query (JOIN):
sql-- Firestore: 2 queries (get user, then get subscription)
-- PostgreSQL: 1 query (JOIN)
SELECT 
    u.id,
    u.email,
    s.status AS subscription_status,
    s.current_period_end
FROM users u
LEFT JOIN subscriptions s ON u.id = s.user_id
WHERE u.email = 'jan@example.com';

3. SUBCOLLECTIONS → FOREIGN KEYS
Firestore:
javascriptusers/{userId}/apiTokens/{tokenId}: {
  token: "encrypted_token",
  name: "My API Key",
  createdAt: Timestamp
}
PostgreSQL:
sqlCREATE TABLE api_tokens (
    id SERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash BYTEA NOT NULL,           -- Hash, nie plaintext!
    name VARCHAR(255),
    last_used_at TIMESTAMP WITH TIME ZONE,
    expires_at TIMESTAMP WITH TIME ZONE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    
    -- Index dla lookup
    CONSTRAINT unique_token_hash UNIQUE (token_hash)
);

CREATE INDEX idx_api_tokens_user_id ON api_tokens(user_id);
Query przykład:
sql-- Pobierz usera z wszystkimi tokenami (1 query!)
SELECT 
    u.id,
    u.email,
    json_agg(
        json_build_object(
            'id', t.id,
            'name', t.name,
            'created_at', t.created_at
        )
    ) AS api_tokens
FROM users u
LEFT JOIN api_tokens t ON u.id = t.user_id
WHERE u.id = 123
GROUP BY u.id, u.email;

4. MAPS/OBJECTS → JSONB
Dla flexible data które nie wymagają queries - użyj JSONB!
Firestore:
javascriptusers/{userId}: {
  email: "jan@example.com",
  metadata: {                    // Flexible map
    preferences: {
      theme: "dark",
      language: "pl"
    },
    lastLogin: Timestamp,
    ipAddress: "192.168.1.1"
  }
}
PostgreSQL:
sqlCREATE TABLE users (
    id SERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    metadata JSONB DEFAULT '{}'::jsonb,  -- ✅ JSONB dla flexible data
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Insert
INSERT INTO users (email, metadata) VALUES (
    'jan@example.com',
    '{"preferences": {"theme": "dark", "language": "pl"}, "lastLogin": "2025-01-15T10:00:00Z"}'::jsonb
);

-- Query JSONB (PostgreSQL supports it!)
SELECT * FROM users 
WHERE metadata->>'preferences'->>'theme' = 'dark';

-- Index JSONB field
CREATE INDEX idx_users_metadata_theme 
    ON users ((metadata->'preferences'->>'theme'));
Kiedy używać JSONB:

✅ Metadata, settings, preferences
✅ Audit logs (flexible structure)
✅ Dynamic fields (różne dla różnych userów)
❌ NIE dla danych które wymagają JOINs
❌ NIE dla relacji (use foreign keys!)


🏢 MULTI-TENANCY (ORGANIZACJE)
Firestore (implicit przez structure):
javascript// Każda organizacja = osobna kolekcja?
organizations/
  {orgId}/
    users/
      {userId}: {...}
    subscriptions/
      {subId}: {...}
PostgreSQL (explicit przez kolumny):
sql-- Tabela organizacji
CREATE TABLE organizations (
    id SERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    slug VARCHAR(100) UNIQUE NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Users należą do organizacji
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    organization_id INTEGER NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
    email VARCHAR(255) NOT NULL,
    name VARCHAR(255),
    role VARCHAR(50) DEFAULT 'user',
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    
    -- Email unique WITHIN organization
    CONSTRAINT unique_email_per_org UNIQUE (organization_id, email)
);

CREATE INDEX idx_users_org_id ON users(organization_id);

-- Subscriptions per organization (nie per user!)
CREATE TABLE subscriptions (
    id SERIAL PRIMARY KEY,
    organization_id INTEGER NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
    stripe_subscription_id VARCHAR(255) UNIQUE,
    status VARCHAR(50) NOT NULL,
    plan VARCHAR(50) NOT NULL,

🔒 ROW-LEVEL SECURITY (RLS)
Automatyczna izolacja danych per organizacja:
sql-- Enable RLS
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
ALTER TABLE subscriptions ENABLE ROW LEVEL SECURITY;

-- Policy: User widzi tylko swoją organizację
CREATE POLICY user_isolation ON users
    FOR ALL
    USING (organization_id = current_setting('app.current_org_id')::INTEGER);

CREATE POLICY subscription_isolation ON subscriptions
    FOR ALL
    USING (organization_id = current_setting('app.current_org_id')::INTEGER);
Użycie w FastAPI:
pythonfrom sqlalchemy import text
from app.db.session import get_db

async def set_current_organization(db: Session, org_id: int):
    """Set RLS context variable"""
    await db.execute(text(f"SET app.current_org_id = {org_id}"))

# W middleware lub dependency
@app.middleware("http")
async def set_rls_context(request: Request, call_next):
    db = request.state.db
    user = request.state.user
    
    if user:
        await set_current_organization(db, user.organization_id)
    
    response = await call_next(request)
    return response

🚀 MIGRATION SCRIPT (Firestore → PostgreSQL)
python# scripts/migrate_from_firebase.py
import asyncio
from google.cloud import firestore
from sqlalchemy.orm import Session
from app.db.session import SessionLocal
from app.db.models import Organization, User, Subscription
from app.core.security import get_password_hash

async def migrate_organizations():
    """Migrate organizations from Firestore to PostgreSQL"""
    
    # Firestore client
    db_firestore = firestore.Client()
    
    # PostgreSQL session
    db_postgres = SessionLocal()
    
    try:
        # Get all organizations from Firestore
        orgs_ref = db_firestore.collection('organizations')
        orgs = orgs_ref.stream()
        
        for org_doc in orgs:
            org_data = org_doc.to_dict()
            
            # Create in PostgreSQL
            org = Organization(
                name=org_data['name'],
                slug=org_data.get('slug', org_doc.id),
                stripe_customer_id=org_data.get('stripeCustomerId'),
                created_at=org_data.get('createdAt')
            )
            db_postgres.add(org)
            db_postgres.flush()  # Get ID
            
            print(f"✅ Migrated org: {org.name} (ID: {org.id})")
            
            # Migrate users for this org
            await migrate_users(db_firestore, db_postgres, org_doc.id, org.id)
            
            # Migrate subscription
            await migrate_subscription(db_firestore, db_postgres, org_doc.id, org.id)
        
        # Commit all changes
        db_postgres.commit()
        print("🎉 Migration completed successfully!")
        
    except Exception as e:
        db_postgres.rollback()
        print(f"❌ Migration failed: {e}")
        raise
    finally:
        db_postgres.close()

async def migrate_users(db_firestore, db_postgres, firebase_org_id, postgres_org_id):
    """Migrate users for one organization"""
    users_ref = db_firestore.collection(f'organizations/{firebase_org_id}/users')
    users = users_ref.stream()
    
    for user_doc in users:
        user_data = user_doc.to_dict()
        
        user = User(
            organization_id=postgres_org_id,
            firebase_uid=user_doc.id,
            email=user_data['email'],
            password_hash=get_password_hash(user_data.get('password', 'RESET_REQUIRED')),
            full_name=user_data.get('name'),
            role=user_data.get('role', 'user'),
            is_active=user_data.get('isActive', True),
            email_verified=user_data.get('emailVerified', False),
            metadata=user_data.get('metadata', {}),
            created_at=user_data.get('createdAt')
        )
        db_postgres.add(user)
        print(f"  ✅ Migrated user: {user.email}")

async def migrate_subscription(db_firestore, db_postgres, firebase_org_id, postgres_org_id):
    """Migrate subscription for one organization"""
    sub_ref = db_firestore.collection(f'organizations/{firebase_org_id}/subscriptions')
    subs = sub_ref.stream()
    
    for sub_doc in subs:
        sub_data = sub_doc.to_dict()
        
        subscription = Subscription(
            organization_id=postgres_org_id,
            stripe_subscription_id=sub_data.get('stripeSubscriptionId'),
            stripe_customer_id=sub_data.get('stripeCustomerId'),
            status=sub_data['status'],
            plan=sub_data['plan'],
            current_period_start=sub_data.get('currentPeriodStart'),
            current_period_end=sub_data.get('currentPeriodEnd'),
            metadata=sub_data.get('metadata', {}),
            created_at=sub_data.get('createdAt')
        )
        db_postgres.add(subscription)
        print(f"  ✅ Migrated subscription: {subscription.plan}")

if __name__ == "__main__":
    asyncio.run(migrate_organizations())

    organization_id INTEGER REFERENCES organizations(id) ON DELETE CASCADE
);

2. Indexy dla często używanych queries:
sql-- ✅ Indexy dla foreign keys
CREATE INDEX idx_users_org_id ON users(organization_id);

-- ✅ Indexy dla lookup fields
CREATE INDEX idx_users_email ON users(email);

-- ✅ Composite index dla multi-column queries
CREATE INDEX idx_subscriptions_org_status ON subscriptions(organization_id, status);
