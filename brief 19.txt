
Brief 19 — Email Templates + Invitations — wersja rozszerzona

Kontekst
Problem A: Maile plain text
Teraz fasthub_core wysyła maile przez EmailTransport (SMTP/Console z Brief 7). Ale treść maila to surowy tekst wklejony w kodzie:
python# Teraz — wszędzie w kodzie:
await send_email(to=email, subject="Reset hasła", body=f"Kliknij link: {url}")
Problemy:

Zero formatowania — klient dostaje goły tekst bez logo, kolorów, przycisków
Treść rozsiana po całym kodzie — zmiana tekstu wymaga szukania po plikach Python
Brak spójności — każdy mail wygląda inaczej
Brak i18n — wszystko po polsku na twardo (a FastHub to boilerplate, może być po angielsku)

Problem B: Brak systemu zaproszeń
Żeby dodać kogoś do organizacji trzeba: ręcznie stworzyć User, ręcznie dodać Member. Nie ma: "Wpisz email → wyślij zaproszenie → osoba klika link → dołącza do firmy z przypisaną rolą".

A) Template Engine — renderowanie szablonów HTML

Kluczowy mechanizm — **nadpisywanie szablonów** przez aplikację:
```
Template Engine szuka szablonu w kolejności:
1. Folder aplikacji (np. autoflow/templates/email/) ← AutoFlow swoje logo, kolory
2. Folder fasthub_core (domyślne szablony)        ← fallback

Tak jak Django szuka templates — najpierw w app, potem w framework.
```

Dzięki temu FastHub daje działające domyślne szablony, a AutoFlow podmienia je na swoje z brandingiem — bez dotykania kodu fasthub_core.

### B) Szablony domyślne (7 maili)

Każdy szablon to para: HTML (ładny) + TXT (fallback dla klientów mailowych bez HTML).

**1. welcome** — po rejestracji
- Kontekst: user_name, org_name, dashboard_url
- Kiedy: po create_user + create_organization

**2. verify_email** — weryfikacja emaila
- Kontekst: user_name, verify_url, expires_in
- Kiedy: po rejestracji lub zmianie emaila

**3. password_reset** — reset hasła
- Kontekst: user_name, reset_url, expires_in
- Kiedy: user klika "zapomniałem hasła"

**4. invitation** — zaproszenie do organizacji
- Kontekst: inviter_name, org_name, role_name, accept_url, expires_in
- Kiedy: admin zaprasza nowego członka

**5. payment_success** — potwierdzenie płatności
- Kontekst: user_name, plan_name, amount, currency, next_billing_date
- Kiedy: hook on_payment_succeeded (Brief 16)

**6. payment_failed** — problem z płatnością
- Kontekst: user_name, plan_name, amount, retry_url, grace_period_days
- Kiedy: hook on_payment_failed (Brief 16)

**7. account_deletion** — potwierdzenie usunięcia konta
- Kontekst: user_name, cancel_url, execute_date, export_url
- Kiedy: DeletionRequest created (Brief 18)

### C) Base Layout — wspólny szablon

Wszystkie maile dziedziczą z base layout (nagłówek z logo, stopka z adresem). Zmiana logo w jednym miejscu → zmienia się we wszystkich mailach.
```
base_layout.html
    ├── header (logo, brand_color, app_name)
    ├── {% block content %}  ← tu wchodzi treść konkretnego maila
    └── footer (company_address, unsubscribe_link, social_links)

welcome.html → {% extends "base_layout.html" %} {% block content %} ... {% endblock %}
D) Brand Config — konfiguracja wyglądu
Zamiast hardkodować kolory i logo w szablonach:
python# Config
EMAIL_BRAND_COLOR: str = "#4F46E5"          # domyślny indigo
EMAIL_BRAND_LOGO_URL: str = ""               # URL do logo
EMAIL_COMPANY_NAME: str = "FastHub"
EMAIL_COMPANY_ADDRESS: str = ""

```

Template engine automatycznie dodaje te zmienne do każdego renderowania. Aplikacja ustawia swoje w .env — zero zmian w kodzie.

### E) Invitation Service — zaproszenia do organizacji

Pełny workflow:

1. Admin wpisuje email + wybiera rolę → `InvitationService.create(email, org_id, role)`
2. System generuje token (UUID), tworzy rekord Invitation, wysyła mail z szablonu "invitation"
3. Osoba klika link → `InvitationService.accept(token)`
4. Jeśli user istnieje → dodaj Member do organizacji
5. Jeśli user NIE istnieje → przekieruj na rejestrację, po rejestracji automatycznie dołącz do organizacji
6. Token wygasa po 7 dniach (konfigurowalne)
7. Admin może anulować zaproszenie lub wysłać ponownie (resend)

A) SendGrid jako EmailTransport backend
Teraz mamy dwa backendy EmailTransport: SMTP i Console. Dodajemy trzeci — SendGrid (API, nie SMTP relay):
EmailTransport(ABC)           ← istniejący kontrakt z Brief 7
    ├── SMTPTransport          ← ✅ jest
    ├── ConsoleTransport       ← ✅ jest  
    └── SendGridTransport      ← NOWY

Config:

    SENDGRID_API_KEY=SG.xxx          ← już jest w config!
    SENDGRID_FROM_EMAIL=noreply@...  ← już jest w config!
SendGrid API to jedno wywołanie HTTP — prostsze niż SMTP (nie trzeba serwera mailowego):
pythonclass SendGridTransport(EmailTransport):
    """
    SendGrid API v3 — wysyłka maili przez REST API.
    
    Docs: https://docs.sendgrid.com/api-reference/mail-send
    Przewaga nad SMTP: nie wymaga serwera mailowego, lepszy deliverability tracking.
    """
    
    async def send(self, to, subject, body, from_email=None, is_html=False):
        async with httpx.AsyncClient() as client:
            response = await client.post(
                "https://api.sendgrid.com/v3/mail/send",
                headers={"Authorization": f"Bearer {self.api_key}"},
                json={
                    "personalizations": [{"to": [{"email": to}]}],
                    "from": {"email": from_email or self.from_email},
                    "subject": subject,
                    "content": [{"type": "text/html" if is_html else "text/plain", "value": body}],
                },
            )
            response.raise_for_status()

B) Sentry — error tracking middleware
Jednolinijkowa integracja z konfigurowalnym DSN. Łapiemy błędy na produkcji zanim klient zgłosi.
python# fasthub_core/middleware/sentry.py

def init_sentry(dsn: str = None, environment: str = "production"):
    """
    Inicjalizacja Sentry — wywołaj raz przy starcie aplikacji.
    
    Użycie:
        # W main.py aplikacji:
        from fasthub_core.middleware.sentry import init_sentry
        init_sentry()  # czyta SENTRY_DSN z config
    """
    try:
        import sentry_sdk
        from sentry_sdk.integrations.fastapi import FastApiIntegration
        from sentry_sdk.integrations.sqlalchemy import SqlalchemyIntegration
        
        _dsn = dsn or get_settings().SENTRY_DSN
        if not _dsn:
            logger.info("Sentry: disabled (no SENTRY_DSN)")
            return
        
        sentry_sdk.init(
            dsn=_dsn,
            environment=environment,
            integrations=[FastApiIntegration(), SqlalchemyIntegration()],
            traces_sample_rate=0.1,  # 10% transakcji
            send_default_pii=False,  # GDPR — nie wysyłaj danych osobowych
        )
        logger.info(f"Sentry: enabled ({environment})")
    except ImportError:
        logger.debug("Sentry: sentry-sdk not installed, skipping")

C) Auto-faktura hook — gotowy plugin "płatność → faktura"
Brief 16 zbudował hook point on_payment_succeeded. Ale każda aplikacja musiałaby sama pisać "pobierz dane firmy, stwórz fakturę, wyślij mailem". To powinien być gotowy komponent:
python# fasthub_core/billing/invoice_hook.py

class AutoInvoiceHook:
    """
    Gotowy hook: po udanej płatności → wystaw fakturę przez Fakturownia.
    
    Użycie:
        handler = StripeWebhookHandler(db)
        handler.on_payment_succeeded = AutoInvoiceHook(db).handle
    
    Albo w registry:
        AutoInvoiceHook.register_for_all_gateways(registry, db)
    
    Wymaga:
        - FAKTUROWNIA_ACCOUNT i FAKTUROWNIA_API_TOKEN w .env
        - Organizacja musi mieć dane do faktury (NIP, adres)
    """
    
    def __init__(self, db):
        self.db = db
    
    async def handle(self, event_data: Dict):
        """
        Po payment_succeeded:
        1. Pobierz tenant z metadata
        2. Pobierz dane organizacji (NIP, adres)
        3. Pobierz plan/addon (nazwa, cena)
        4. Stwórz fakturę przez FakturowniaClient
        5. Wyślij mailem do kupującego
        6. Zapisz numer faktury w BillingEvent
        """
        tenant_id = event_data.get("tenant_id")
        if not tenant_id:
            return
        
        # Pobierz org
        org = await self._get_organization(tenant_id)
        if not org or not org.nip:
            logger.warning(f"Cannot issue invoice: org {tenant_id} has no NIP")
            return
        
        # Stwórz fakturę
        from fasthub_core.clients import FakturowniaClient
        client = FakturowniaClient.from_config()
        
        invoice = await client.create_invoice({
            "kind": "vat",
            "buyer_name": org.name,
            "buyer_tax_no": org.nip,
            "buyer_email": org.email,
            "buyer_street": org.billing_street,
            "buyer_city": org.billing_city,
            "buyer_post_code": org.billing_postal_code,
            "positions": [{
                "name": event_data.get("plan_name", "Subskrypcja"),
                "quantity": 1,
                "total_price_gross": event_data.get("amount", 0) / 100,
                "tax": 23,  # VAT 23%
            }],
        })
        
        # Wyślij mailem
        if invoice.get("id"):
            await client.send_invoice_by_email(invoice["id"])
        
        return invoice
