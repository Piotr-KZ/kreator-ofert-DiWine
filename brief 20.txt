Brief 20 — Polskie bramki płatnicze — wersja rozszerzona

Kontekst
Brief 16 zbudował fundament — PaymentGateway(ABC), PaymentGatewayRegistry, StripeGateway. Ale Stripe w Polsce to nisza. Polskie firmy używają PayU, Tpay, Przelewy24. Klient końcowy chce płacić BLIKiem, przelewem z banku, kartą Visa — nie "Stripe checkout".
Brief 20 implementuje bramki które realnie obsługują 95% polskiego rynku płatności online.

Kluczowa różnica: subskrypcje
Stripe ma natywne subskrypcje — mówisz "pobieraj 199 zł co miesiąc" i Stripe sam to robi.
Polskie bramki tego NIE MAJĄ. PayU, Tpay, P24 obsługują jednorazowe płatności. Żeby mieć subskrypcję trzeba:

Co miesiąc (cron) generować nową płatność
Wysłać klientowi link lub obciążyć zapisaną kartę
Sprawdzić czy zapłacił
Jeśli nie → grace period → powiadomienie → jeśli dalej nie → cancel

To wymaga nowego komponentu: RecurringManager.
Stripe:
    create_subscription() → Stripe sam pobiera co miesiąc → webhook "payment_succeeded"

Polskie bramki:
    create_subscription() → zapisz w bazie "plan Pro, 199 PLN, co miesiąc"
    RecurringManager (cron co godzinę):
        → znajdź subskrypcje gdzie current_period_end < teraz
        → dla każdej: create_payment() przez bramkę
        → czekaj na webhook "zapłacone"
        → jeśli zapłacone: przedłuż period o miesiąc
        → jeśli nie zapłacone po 3 dniach: status=past_due, wyślij mail
        → jeśli nie zapłacone po 14 dniach: status=canceled, downgrade do free

Jak działa każda bramka
PayU
Flow: Redirect. Tworzymy "order" przez API → dostajemy URL → klient płaci na stronie PayU → PayU wysyła notify na nasz URL.
API: https://secure.payu.com/api/v2_1/orders
Auth: OAuth2 (client_id + client_secret → bearer token)
Tworzenie: POST /orders → redirectUri (tam odsyłamy klienta)
Notify: POST na nasz endpoint z JSON (status: COMPLETED/CANCELED/...)
Weryfikacja: signature w nagłówku (MD5 lub SHA256)
Config:
PAYU_POS_ID=123456
PAYU_MD5_KEY=abc123...
PAYU_CLIENT_ID=123456
PAYU_CLIENT_SECRET=xyz789...
PAYU_ENVIRONMENT=sandbox|production  # sandbox.payu.com vs secure.payu.com
Metody płatności PayU: BLIK (blik), karty (c), przelewy bankowe (pbl), Google Pay (jp), Apple Pay (ap), raty (ai).
Tpay (Autopay)
Flow: Redirect. Tworzymy transakcję → dostajemy URL → klient płaci → Tpay wysyła callback.
API: https://api.tpay.com/ (REST, nowe API)
Auth: client_id + client_secret → OAuth2 bearer
Tworzenie: POST /transactions → transactionPaymentUrl
Callback: POST na nasz endpoint z danymi transakcji
Weryfikacja: checksum (MD5 z kodem bezpieczeństwa)
Config:
TPAY_CLIENT_ID=xxx
TPAY_CLIENT_SECRET=xxx
TPAY_SECURITY_CODE=xxx
TPAY_ENVIRONMENT=sandbox|production
Przelewy24
Flow: Redirect. Rejestrujemy transakcję → dostajemy token → redirect na P24 → klient płaci → P24 wywołuje callback → my weryfikujemy transakcję.
API: https://secure.przelewy24.pl/api/v1/
Auth: Basic (merchantId + CRC key)
Rejestracja: POST /transaction/register → token
Redirect: https://secure.przelewy24.pl/trnRequest/{token}
Callback: POST na nasz endpoint
Weryfikacja: PUT /transaction/verify (OBOWIĄZKOWE — bez tego P24 nie uznaje)
Config:
P24_MERCHANT_ID=123456
P24_POS_ID=123456         # zwykle = MERCHANT_ID
P24_CRC_KEY=abc123...
P24_ENVIRONMENT=sandbox|production
Ważne: P24 jako jedyna WYMAGA weryfikacji po callbacku. Po otrzymaniu notify musisz wywołać /transaction/verify — dopiero wtedy transakcja jest potwierdzona.
PayPal
Flow: Redirect. Tworzymy order → redirect na PayPal → klient płaci → capture.
API: https://api-m.paypal.com/v2/
Auth: OAuth2 (client_id + client_secret → bearer)
Tworzenie: POST /checkout/orders → approval URL
Po redirect: POST /checkout/orders/{id}/capture
Webhook: POST na nasz endpoint z event
Config:
PAYPAL_CLIENT_ID=xxx
PAYPAL_CLIENT_SECRET=xxx
PAYPAL_ENVIRONMENT=sandbox|production
PayPal MA natywne subskrypcje (Billing Plans), więc RecurringManager nie jest potrzebny dla PayPala — analogicznie do Stripe.

RecurringManager — serce polskich subskrypcji
pythonclass RecurringManager:
    """
    Zarządza cyklem subskrypcji dla bramek BEZ natywnych subskrypcji.
    
    Uruchamiany jako cron job (Brief 15 task queue):
    - Co godzinę: sprawdź subskrypcje do odnowienia
    - Generuj płatności
    - Obsłuż grace period
    - Downgrade jeśli nie zapłacono
    
    NIE dotyczy Stripe i PayPal (mają natywne subskrypcje).
    DOTYCZY: PayU, Tpay, P24, iMoje, CashBill.
    """
    
    async def process_renewals(self, db):
        """
        Główna metoda — wywoływana co godzinę przez cron.
        
        1. Znajdź subskrypcje gdzie:
           - status = active
           - gateway != stripe, paypal (te mają natywne)
           - current_period_end < now
        
        2. Dla każdej:
           a) Jeśli klient ma zapisaną kartę → obciąż automatycznie
           b) Jeśli nie → wyślij mail z linkiem do płatności
        
        3. Obsłuż grace period:
           - 0-3 dni po terminie: status=active, wyślij reminder
           - 3-14 dni: status=past_due, wyślij ostrzeżenie
           - 14+ dni: status=canceled, downgrade do free
        """
        ...
    
    async def create_renewal_payment(self, subscription, db):
        """Stwórz płatność odnowieniową przez bramkę subskrypcji."""
        registry = get_payment_registry()
        result = await registry.create_payment(
            gateway_id=subscription.gateway_id,
            amount=subscription.amount,
            currency=subscription.currency,
            description=f"Odnowienie: {subscription.plan_name}",
            return_url=f"{FRONTEND_URL}/billing/renewal-success",
            metadata={"tenant_id": subscription.tenant_id, "renewal": True},
        )
        return result
    
    async def handle_renewal_paid(self, subscription, db):
        """Po zapłaceniu — przedłuż okres o miesiąc/rok."""
        if subscription.billing_interval == "monthly":
            subscription.current_period_end += timedelta(days=30)
        else:
            subscription.current_period_end += timedelta(days=365)
        subscription.status = "active"
        ...
    
    async def handle_grace_period(self, subscription, days_overdue, db):
        """Logika grace period."""
        if days_overdue <= 3:
            await send_templated_email("payment_reminder", ...)
        elif days_overdue <= 14:
            subscription.status = "past_due"
            await send_templated_email("payment_failed", ...)
        else:
            subscription.status = "canceled"
            await self._downgrade_to_free(subscription, db)
            await send_templated_email("subscription_canceled", ...)

PaymentGatewayRegistry — aktualizacja from_config
python@classmethod
def from_config(cls) -> "PaymentGatewayRegistry":
    registry = cls()
    
    # Próbuj każdą bramkę — jeśli ma klucze w env, zarejestruj
    gateway_classes = [
        ("fasthub_core.billing.gateways.stripe_gateway", "StripeGateway"),
        ("fasthub_core.billing.gateways.payu_gateway", "PayUGateway"),
        ("fasthub_core.billing.gateways.tpay_gateway", "TpayGateway"),
        ("fasthub_core.billing.gateways.p24_gateway", "P24Gateway"),
        ("fasthub_core.billing.gateways.paypal_gateway", "PayPalGateway"),
    ]
    
    for module_path, class_name in gateway_classes:
        try:
            module = importlib.import_module(module_path)
            gateway_class = getattr(module, class_name)
            gateway = gateway_class()
            registry.register(gateway)  # register sprawdza is_configured()
        except ImportError:
            pass
    
    return registry
